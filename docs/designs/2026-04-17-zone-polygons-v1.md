# Design: Zone Polygons v1 — 행정동 union 기반 정확도 리팩토링

Generated inline by `/plan-eng-review` on 2026-04-17 (no prior CEO plan; promoted directly from research session).
Repo: ORANKWON/korea-robotaxi-dashboard

## 컨텍스트

현재 상태:
- `data/zones.json` = 29개 시범운행지구. 각 zone에 `lat/lng/area_km2/boundary: [[lat,lng], ...]` 필드.
- boundary는 **수작업으로 그린 다각형** (강남 zone은 33점, 다른 zone들은 12-42점 타원/다각형).
- 강남 zone 검증 결과: 면적 18.62 km² (공식 20.4 km² 대비 -8.7% 오차). 시각적으로 image 비교했을 때도 미세하게 어긋남.
- 다른 zone들은 검증조차 안 됨. **타원형 boundary는 행정구역과 일치할 수 없음** (행정구역은 항상 비-타원).

핵심 문제:
1. **데이터 출처 부재**: 국토부/지자체가 시범운행지구 GeoJSON/SHP를 공개하지 않음. 고시문 PDF만 존재 (예: "○○동 일원").
2. **수작업 한계**: 한 zone당 30-50개 좌표 수동 입력 → 29개 zone = 최대 1,450개 좌표. 휴먼 에러 확률 100%.
3. **면적 검증 안 됨**: `area_km2` 필드는 공식 발표값을 복붙. 실제 boundary로 계산하면 ±10% 오차.
4. **유지보수 불가**: 새 zone 추가 시 1시간+. 경계 변경 시 수정 어려움.

해결 방향: **행정동 GeoJSON union**. 한국 행정동 경계는 `vuski/admdongkor` (MIT, EPSG:4326, 2025-08-05 기준) 무료 공개. 시범운행지구는 항상 행정동 단위 또는 그 일부 → 행정동 코드 리스트로 정의 → turf.union으로 polygon 자동 생성.

## 1순위 사용자
지도 페이지(`/map`) 사용자 + companies.json `zones` 필드 참조하는 UI 전체. 정확한 경계 = 신뢰도.

## Approach

**Build-time 산출물 모델.** 기존 zones.json은 source of truth + build artifact가 섞여 있음. 분리:
- **Source of truth (수동):** `data/zones.source.json` (id, name, lat, lng, area_km2, status, companies, description, designated, **dong_codes**)
- **Build artifact (자동):** `data/zones.json` (위 + boundary + boundary_source + boundary_built_at)
- **Build script:** `scripts/build-zones.ts` — `bun run build:zones`로 실행
- **CI gate:** `bun run check:zones` → diff 있으면 fail (lint과 동일 패턴)

빌드 스크립트가 하는 일:
1. `data/admin-dong.geojson` 로드 (vuski 데이터, 행정동 ~3,500개)
2. 각 zone의 dong_codes에 매칭되는 features 필터
3. `@turf/turf` `union()` 으로 polygon 합치기
4. `simplify(0.0001)` 로 좌표 수 절반 정도로 단축 (Visvalingam 알고리즘 in turf)
5. `boundary` 필드 = `[lng, lat]` GeoJSON → `[lat, lng]` Leaflet 변환
6. `area()` 계산 → `area_km2_computed` 필드. 공식값과 ±5% 초과 시 WARN
7. atomic write

데이터 소스:
- `vuski/admdongkor` (https://github.com/vuski/admdongkor) MIT, 일별 업데이트, EPSG:4326, GeoJSON
- 파일: `HangJeongDong_ver20240805.geojson` (5MB, 3,500 features)
- properties: `adm_cd` (행정동 코드 10자리), `adm_nm` (이름 "서울특별시 강남구 역삼1동")

## Bundle 정의

### Phase 1: 인프라 — 1h CC
- `package.json`에 `@turf/turf` 의존성 추가 (`bun add @turf/turf`)
- `data/admin-dong.geojson` 다운로드 (vuski raw URL → curl → checksum 검증 → commit)
- `scripts/build-zones.ts` 작성 (~150 LoC)
- `package.json` scripts: `"build:zones": "bun scripts/build-zones.ts"`, `"check:zones": "bun scripts/build-zones.ts --check"`
- `data/zones.source.json` 생성 = 현재 zones.json의 수동 필드 추출
- `data/zones.json` = build artifact 표시 주석 추가 (gitignore X — vercel 빌드가 인라인으로 읽음)

### Phase 2: 강남 zone (id:1) 마이그레이션 — 0.5h CC
- 강남구청 행정동 지도 + 시범운행지구 고시문 대조 → 16개 dong_codes 결정
- `dong_codes`만 `zones.source.json`에 입력
- `bun run build:zones` 실행 → 새 boundary 생성
- 면적 검증: 공식 20.4 km² vs computed → ±5% pass
- 시각 검증: `bun run dev` → /map 에서 강남 zone 확인 → 첨부 image 비교
- diff PR: zones.source.json + zones.json 동시 변경 (생성된 boundary는 review용 inline)

### Phase 3: 나머지 28 zones 마이그레이션 — 4h CC
- 우선순위 (사용자 노출 빈도 기준):
  1. **운행 중 (사용자 매일 봄):** 강남(완), 청계천, 상암, 마포, 세종 → 4개
  2. **시범운행 활발:** 대구, 안양, 제주, 대전 → 4개
  3. **나머지 시범운행지구** → 20개
- 각 zone당 평균 8-15분 (dong_code 결정이 대부분 시간)
- 각 zone 마이그 후 `--check` 모드로 area diff 출력
- area diff >5% 인 zone은 별도 issue 또는 dong_codes 재검토

### Phase 4: 검증 + CI 자동화 — 1h CC
- `validate_data.py` 확장:
  - boundary가 dong_codes 기반인지 확인 (boundary_source 필드 존재)
  - area_km2_computed가 area_km2와 ±10% 이내인지 확인
- `.github/workflows/ci.yml` 추가:
  - `bun run check:zones` 단계 → 빌드 산출물 drift 감지
  - source 변경 시 build:zones 자동 실행 + commit
- `tests/test_build_zones.ts` (vitest):
  - 단일 dong → boundary == dong polygon
  - 여러 dong union 결과의 area = 각 dong area 합과 ±1% 이내
  - 누락된 adm_cd → 명확한 에러 메시지
  - dong_codes 빈 배열 → boundary 생성 skip + WARN
  - 인접하지 않은 dong들 union → MultiPolygon 결과 → leaflet 호환 확인

### Phase 5: 지도 페이지 통합 — 0.5h CC
- `MapView.tsx` 검증: MultiPolygon boundary 렌더링 (현재 단일 다각형 가정)
- zone hover 시 "행정동: 역삼1동, 역삼2동, ..." 툴팁 추가 (선택)
- 지도 페이지 푸터에 데이터 출처 명시: "경계: vuski/admdongkor (2025-08-05) 행정동 union"

총: ~7h CC (= 인간 팀 ~2.5일)

## Edge Cases

| 상황 | 처리 |
|------|------|
| dong_codes에 vuski 데이터에 없는 코드 | build 스크립트 명확한 에러 + zone id 출력 + 종료 코드 1 |
| dong_codes 빈 배열 또는 미정의 | boundary skip + WARN + 기존 boundary 보존 (마이그 중인 zone 보호) |
| 인접하지 않은 dong들 (예: 제주공항+중문) | turf.union이 MultiPolygon 반환 → leaflet `Polygon` 컴포넌트로는 첫 polygon만 렌더 → `polygon` array로 변환 필요 |
| vuski 데이터 업데이트로 행정동 경계 변경 | `boundary_built_at` + git diff로 감지. 의도 외 변경 시 review |
| area_km2 공식값 vs computed > 5% | WARN, 가능 원인: (a) zone이 일부 dong만 포함, (b) 공식값이 부정확, (c) dong_codes 누락 |
| 시범운행지구가 행정동 단위가 아닌 경우 (도로 한 쪽만 등) | 현실: 거의 없음. 있다면 fallback으로 수동 boundary 유지 (dong_codes 비움 + boundary 직접 입력) |
| 강남 zone 16개 dong + "서초동 일부" | 서초동 전체 포함 후 area mismatch 수용 OR 서초동 제외 + area 약간 작게. 1차는 전체 포함. |
| vuski raw URL 다운 | data 파일은 이미 commit됨. 재다운로드는 reproducibility용. |
| MultiPolygon 시 leaflet z-index 이슈 | 한 polygon씩 별도 Polygon 컴포넌트로 렌더 |
| zones.json 직접 수정한 PR이 옴 | CI `check:zones` 가 fail → `bun run build:zones` 실행 후 재커밋 안내 |

## Risks

| Risk | Mitigation |
|------|-----------|
| vuski 데이터 라이선스/유지보수 변경 | MIT, repo는 일별 업데이트 활발. 1년 stale 대응으로 fork 고려 (TODO) |
| 행정동 코드 (10자리) 결정에 시간 소요 | vuski geojson에 한글 이름 있음 → "역삼1동" → 검색하면 코드 즉시. zone당 5-10분. |
| 5MB geojson commit이 repo bloat | git LFS 또는 `--depth=1` 클론. 5MB는 acceptable (image보다 작음). |
| turf union 성능 | 행정동당 ~100점, 16개 union = ~1,600점 처리. <100ms. 29 zones 전체 빌드 = <5초. |
| MultiPolygon 렌더 깨짐 | Phase 5에서 MapView 검증 + 테스트 |
| zone area_km2 공식값과 computed 항상 mismatch | 공식값은 마케팅 round 가능성. mismatch가 5% 이내면 WARN 무시, 그 외는 dong_codes 재검토. |
| 마이그 중 일부 zone만 dong_codes 있는 hybrid 상태 | build 스크립트가 dong_codes 없는 zone은 기존 boundary 보존. 부분 마이그레이션 안전. |

- `area_km2_computed` vs `area_km2` 평균 오차: **±10% → ±2%**

- 시범운행지구 PDF 고시문 자동 파싱 (수동 dong_code 입력으로 충분)
